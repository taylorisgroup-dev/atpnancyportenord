# Handoff Report

## Observation
The auditor report identified three primary Integrity Violations in the Tier 1 E2E tests:
1. `5-cv-extraction-api.spec.ts` uses an array of accepted HTTP status codes (`expect([200, 500]).toContain(response.status())`) and conditional assertions, effectively allowing `500 Internal Server Error` to pass silently. It also uses a deliberately invalid PDF payload (`dummy-pdf-content`) which triggers backend errors.
2. `6-seo-metadata.spec.ts` utilizes wildcard regular expressions (e.g., `/.+/`) to check for the mere existence of SEO metadata attributes rather than their actual correctness.
3. `4-admin-mutation.spec.ts` performs stateful interactions (creating, editing, and deleting admin data) using hardcoded IDs and lacks any state teardown, which causes test flakiness across multiple runs and pollutes the environment.

## Logic Chain
1. **CV Extraction API**: Allowing 500 errors creates a "Self-certifying test" that masks backend failures. The fix requires asserting exactly `200` for success cases, providing a truly valid, minimal PDF file payload, and explicitly mocking the backend or external Gemini API if timeouts/errors need to be explicitly tested.
2. **SEO Metadata**: Wildcard assertions circumvent correctness testing. To fix this, assertions must compare against the exact strings expected to be generated by the application (e.g., `'ATP Nancy - Accueil'`) or rely on a deterministic, isolated data set.
3. **Admin Mutations**: Stateful tests that rely on shared state (e.g., editing 'item-1') will fail if run concurrently or repeatedly. The tests must be self-contained: any entity created or mutated must be explicitly set up beforehand and cleaned up (deleted) afterward via `test.afterEach` or inline teardowns.

## Caveats
- Exact expected SEO metadata text is not currently documented in this report; the implementer will need to extract the actual expected titles and descriptions from the application source.
- The `admin-mutation` teardown strategy might require additional helper functions to clean up state via direct API calls rather than the UI to ensure robust cleanup.

## Conclusion
The Tier 1 test suite requires an immediate rewrite to enforce strict testing standards.
- `5-cv-extraction-api.spec.ts`: Must assert exact HTTP status codes (200 for success, 504/400 for specific errors) and remove conditional assertions. Test payloads must be structurally valid or explicitly mocked.
- `6-seo-metadata.spec.ts`: Must replace `/.+/` wildcards with exact expected values.
- `4-admin-mutation.spec.ts`: Must implement explicit creation and teardown phases for every mutated entity to avoid test pollution.

## Proposed Test Case Designs

### 1. `5-cv-extraction-api.spec.ts`
- **Test**: `should process a valid CV PDF and return structured JSON`
  - **Logic**: Post a structurally valid minimal PDF buffer. Assert `expect(response.status()).toBe(200)`. Assert `body.skills` and `body.experience` exist unconditionally.
- **Test**: `should handle Gemini API timeout/failure gracefully`
  - **Logic**: Mock the external Gemini API call using Playwright's `route` to force a timeout. Assert `expect(response.status()).toBe(504)` strictly.

### 2. `6-seo-metadata.spec.ts`
- **Test**: `should have correct title and description on Home page`
  - **Logic**: Assert exact title (e.g., `expect(page).toHaveTitle('ATP Nancy - Accueil')`) and exact description string.
- **Test**: `should have correct Open Graph tags on Home page`
  - **Logic**: Assert `og:title` is strictly equal to the expected site title.
- **Test**: `should have dynamic SEO metadata on specific Job Offer page`
  - **Logic**: Mock or seed a specific job offer (e.g., 'Développeur Web'). Assert the title is exactly the dynamically generated title (e.g., `'Développeur Web | ATP Nancy'`).

### 3. `4-admin-mutation.spec.ts`
- **Test**: `should add a new Job Offer and clean it up`
  - **Logic**: Navigate to new job offer form, fill and submit. Assert it appears. In teardown, delete the specific job offer created.
- **Test**: `should edit an existing Agenda event`
  - **Logic**: Setup: Create a temporary event. Act: Edit this newly created event. Assert: Check the update. Teardown: Delete the event.
- **Test**: `should delete an Organigramme member`
  - **Logic**: Setup: Create a temporary member. Act: Delete the member and assert it is removed. Teardown: Fallback in `afterEach` to ensure it is deleted if the test fails midway.

## Verification Method
1. Run `npx playwright test tests/e2e/tier1/5-cv-extraction-api.spec.ts`. The test MUST fail if the backend is broken (returns 500).
2. Run `npx playwright test tests/e2e/tier1/6-seo-metadata.spec.ts`. Modify the application's SEO title; the test MUST fail, proving it checks for correctness.
3. Run `npx playwright test tests/e2e/tier1/4-admin-mutation.spec.ts` multiple times sequentially. It MUST pass every time without data collision. Check the database/UI to ensure no leftover test data remains.
